JUNIOR ALADDIN DASHBOARD ROADMAP
Obsidian Night / Institutional Terminal Edition

SOURCE ANCHORS
===============================================================================
This document is aligned to:
1. system plan and roadmap.txt
2. main plan and roadmap.pdf
3. ChatGPT dash.png

The design target is not a generic admin dashboard. It is an institutional
trading terminal inspired by Bloomberg, Reuters Eikon, and sovereign command
centers. The visual language must preserve the deep-night, cyan-accent,
high-density cockpit seen in the image while becoming sharper, more legible,
more deterministic, and more useful for a junior developer to build step by step.

PRIMARY RULE
===============================================================================
The dashboard never creates alpha. It only reveals alpha, protects the operator,
and exposes the backend state with absolute fidelity.

If a field is not already known by the backend, the dashboard must show it as
missing, stale, or disabled. It must never guess.

SYSTEM INTENT
===============================================================================
Junior Aladdin is a 16-layer sovereign trading stack. The dashboard is Layer 15C
and the visibility layer for the entire machine. It is a read-only control tower
with command handshakes for safe mode changes, live enablement, and emergency
containment.

The dashboard must reflect all 16 backend layers:
1. External Data Universe
2. Data Validation and Ingestion
3. Feature Computation
4. Narrative Engine
5. Regime Engine
6. Time Context Engine
7. Market DNA Fingerprint Engine
8. Four-Brain Orchestrator
9. Strategy Engine
10. Trap Detection Engine
11. Dynamic Confluence Scoring Engine
12. ML / AI Filter Layer
13. Behavioral Sentinel
14. Portfolio Risk Engine
15. Captain Orchestrator, Position Management, Compliance, Journal, Self-Learning
16. Dashboard Surface and Operator Command Plane

The dashboard must expose these layers as live panels, charts, alerts, and
operator actions without duplicating backend logic.

SECTION 2: SYSTEM ARCHITECTURE ALIGNMENT
===============================================================================

2.1 DASHBOARD POSITION IN THE 16-LAYER STACK
-------------------------------------------------------------------------------
The dashboard reads a snapshot of backend state and renders it into:
- panels
- gauges
- charts
- logs
- command dialogs
- kill-switch surfaces
- compliance overlays

It must never re-compute core alpha logic. If a field is already computed in the
backend, the dashboard only visualizes it.

2.2 PYQT6 EVENT LOOP MODEL
-------------------------------------------------------------------------------
The dashboard runtime shall use a single QApplication event loop with three
independent update rhythms:

HOT loop:
- frequency: 100 to 250 ms
- purpose: system health, positions summary, live risk, clock, kill-switch
- implementation: QTimer in the GUI thread consuming prebuilt hot frames

WARM loop:
- frequency: 1000 ms
- purpose: most panel content, narrative, regime, permissions, logs, alerts
- implementation: queued signal update from a worker thread

COLD loop:
- frequency: 5000 ms to 30000 ms depending on payload
- purpose: history tables, ML explainability, journal preview, diagnostics
- implementation: batch refresh or lazy load

The event loop must stay deterministic:
1. read kill-switch first
2. decode latest valid snapshot
3. apply diff to shared state
4. update hot widgets
5. schedule warm widgets
6. schedule cold widgets only if visible

2.3 SNAPSHOT TIERS
-------------------------------------------------------------------------------
HOT tier:
- latency budget: under 50 ms end to end
- payload size target: under 8 KB
- format: binary frame with fixed header + compact binary payload
- contents: clock, mode, feed health, LTP, P&L, risk state, active positions,
  regime flag, captain state, critical alerts

WARM tier:
- latency budget: under 250 ms
- payload size target: under 128 KB
- format: binary payload or compressed JSON
- contents: candles, features, option chain summary, narrative, regime,
  opportunities, brain status, session context

COLD tier:
- latency budget: under 5 seconds
- payload size target: under 1 MB after compression
- format: compressed JSON or msgpack bundle
- contents: full history, logs, journal entries, SHAP snippets, replay state

2.4 BINARY SERIALIZATION FOR HOT TIER
-------------------------------------------------------------------------------
The HOT tier must use a binary envelope so the GUI can decode a frame quickly
without allocating large Python objects.

Recommended wire format:

+------------+------------+------------+------------+------------+------------+
| MAGIC(4)   | VERSION(2) | KIND(2)    | SEQ(8)     | TS(8)      | FLAGS(4)   |
+------------+------------+------------+------------+------------+------------+
| PAYLOAD_LEN(4) | CRC32(4) | PAYLOAD(N bytes)                                |
+-----------------+---------+--------------------------------------------------+

Payload encoding:
- msgpack for compact structured values
- optional zstd compression when payload exceeds a threshold
- field names are stable and versioned
- numeric arrays should be sent as packed lists, not nested dicts, when the
  receiving widget only needs sequential values

Validation steps:
1. verify magic
2. verify version
3. verify payload length
4. verify CRC32
5. discard stale sequence numbers
6. reject frames older than the last accepted timestamp

The dashboard must keep the last valid hot frame so the UI can degrade gracefully
instead of blanking out on a temporary bad packet.

2.5 SHARED MEMORY KILL-SWITCH
-------------------------------------------------------------------------------
The kill-switch must live in a tiny shared memory block or equivalent atomic
control channel that both backend and dashboard can read with near-zero latency.

Control block contract:
- version
- backend pid
- dashboard pid
- mode flag
- emergency stop flag
- live approval flag
- compliance state
- heartbeat timestamp
- last acknowledged sequence
- safe-lock reason code

Required behavior:
- if heartbeat is stale beyond threshold, dashboard shows SAFE and locks live
  enablement dialogs
- if emergency stop is set, all live control surfaces turn red immediately
- if compliance state is invalid, LIVE mode cannot be enabled
- if shared memory cannot be read, dashboard enters conservative display mode

2.6 DATA FLOW DIAGRAM
-------------------------------------------------------------------------------

BACKEND
  |  MarketState snapshot
  v
Snapshot Packager
  |  HOT/WARM/COLD frames
  v
Shared Memory / IPC Queue
  |  binary frames
  v
Dashboard Decoder
  |  validated state diff
  v
Widget Model -> Panels -> Charts -> Alerts

The dashboard only renders state that has already passed backend validation.

SECTION 5: COMMAND AND CONTROL
===============================================================================

5.1 LIVE MODE THREE-STEP HANDSHAKE
-------------------------------------------------------------------------------
LIVE mode is not a button. It is a compliance handshake.

Step 1: Identity verification screen
- confirm broker connection
- confirm user role or operator profile
- confirm static IP is the approved one
- confirm backend PID and dashboard PID are both healthy

Step 2: Compliance verification screen
- confirm Algo-ID is valid
- confirm limit orders only mode is available
- confirm emergency stop channel is working
- confirm audit trail sink is writable

Step 3: Live enable confirmation screen
- user types a short confirmation phrase
- dashboard receives a backend signed token with TTL
- backend flips mode only if kill-switch, compliance, and health all pass

The dashboard must visually lock LIVE mode until all three steps succeed.

5.2 MODE SURFACES
-------------------------------------------------------------------------------
- ALERT: read-only, no live order capability, safest first mode
- PAPER: live data, simulated execution
- LIVE: real execution, strict compliance gates

5.3 EMERGENCY STOP
-------------------------------------------------------------------------------
Emergency stop is always visible.
It must:
- be one click or one hotkey away
- require no scrolling
- turn the terminal red and freeze LIVE enablement
- preserve current state for audit after activation

===============================================================================
SECTION 6: CHARTING ENGINE
===============================================================================

6.1 QWEBENGINE + PLOTLY BRIDGE
-------------------------------------------------------------------------------
Plotly must run inside a QWebEngineView and never be reloaded per tick.
The page is loaded once, then updated by JavaScript patches.

Bridge contract:
- Python prepares a compact frame
- the bridge emits a JavaScript payload
- Plotly receives incremental updates
- the chart maintains its own ring buffers in browser memory

6.2 EXTENDTRACES STRATEGY FOR 5000+ TICKS
-------------------------------------------------------------------------------
The chart must not freeze when the tick count passes 5000.

Required method:
- initialize traces once
- use Plotly.extendTraces for live points
- limit each visible trace with maxpoints
- batch incoming points into one animation frame
- never call full Plotly.newPlot on every update

Anti-freeze rules:
1. batch tick writes into 100 ms or 250 ms windows
2. keep only visible window points in the DOM layer
3. store history in Python or backend memory, not in the browser DOM
4. use requestAnimationFrame or equivalent timing in JS
5. if the chart falls behind, coalesce frames instead of queueing all of them

6.3 PLOTLY DATA POLICY
-------------------------------------------------------------------------------
- candlestick chart traces are primary
- volume histogram is secondary
- overlays for VWAP, OR, IB, SMC, liquidity, and component guard must be thin
- tooltips must be informative but not cluttered
- no animated rainbow indicators

===============================================================================
SECTION 7: REALTIME PERFORMANCE CONTRACT
===============================================================================

7.1 LATENCY BUDGETS
-------------------------------------------------------------------------------
- hot state decode: under 10 ms
- hot widget patch: under 20 ms
- warm panel refresh: under 100 ms
- chart delta apply: under 30 ms for a small batch
- live mode handshake: under 2 seconds including user confirmation

7.2 GRACEFUL DEGRADATION
-------------------------------------------------------------------------------
If the backend pauses or feed quality drops:
- freeze warm and cold widgets first
- keep hot status alive longer than everything else
- retain last valid chart frame
- show stale badges instead of blanking panels
- preserve the last successful kill-switch state

7.3 MEMORY POLICY
-------------------------------------------------------------------------------
- avoid loading full historical arrays into every widget
- use shared projections, not duplicate copies
- panel-local caches must be bounded
- chart history must be ring-buffer based
